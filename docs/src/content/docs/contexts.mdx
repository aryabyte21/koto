Context profiles are koto's core differentiator. They let you define different translation tones, terminology, and instructions for different parts of your application. Instead of treating all strings identically, koto adapts its translations to match the intent of each section.

## Why context profiles matter

Consider the word "Submit" in two different places:

- On a **payment form**, it should translate as the formal equivalent of "Confirm Payment" in each language.
- On a **feedback widget**, it could translate as the casual equivalent of "Send" or "Share."

Without context, a translation tool picks one and applies it everywhere. With koto's context profiles, each usage gets the right translation automatically.

## Defining context profiles

Context profiles are defined in the `contexts` section of your `koto.config.ts`. Each profile specifies which files it applies to, the desired tone, and any additional instructions or glossaries.

```ts
import { defineConfig } from 'koto'

export default defineConfig({
  sourceLocale: 'en',
  targetLocales: ['es', 'fr', 'de', 'ja'],
  files: ['src/locales/[locale].json'],
  provider: {
    name: 'openai',
    model: 'gpt-4o-mini',
  },

  contexts: {
    // Applied to files not matched by any other context
    default: {
      tone: 'concise',
      instructions: 'UI strings. Keep translations short and consistent with platform conventions.',
    },

    marketing: {
      tone: 'persuasive and energetic',
      instructions: 'Keep translations punchy. Prefer short, impactful words. Maintain brand voice.',
      files: ['src/locales/marketing/[locale].json'],
    },

    legal: {
      tone: 'formal and precise',
      instructions: 'Use jurisdiction-appropriate legal terminology. Avoid ambiguity.',
      glossary: './glossaries/legal.json',
      files: ['src/locales/legal/[locale].json'],
    },

    support: {
      tone: 'friendly and helpful',
      instructions: 'Use clear, simple language. Be empathetic. Avoid jargon.',
      files: ['src/locales/support/[locale].json'],
    },
  },
})
```

## How context resolution works

<Steps>

1. **File matching** — When koto translates a file, it checks which context profile's `files` pattern matches the file path.

2. **Default fallback** — If no context matches, the `default` context is applied. If no `default` context is defined, koto translates without additional context.

3. **Instructions assembly** — koto builds a prompt that includes the context's `tone`, `instructions`, and any `glossary` terms. This prompt is sent to the LLM alongside the source strings.

4. **Translation** — The LLM translates the strings with full awareness of the intended context, producing translations that match the desired tone and terminology.

</Steps>

## Context profile options

### `tone`

**Type:** `string`

A natural language description of the desired tone. This is included in the LLM prompt to guide translation style.

```ts
tone: 'concise'
tone: 'formal and precise'
tone: 'persuasive and energetic'
tone: 'friendly, casual, and approachable'
```

### `instructions`

**Type:** `string`

Additional instructions for the LLM. Use this for domain-specific guidance, formatting rules, or constraints.

```ts
instructions: 'UI strings. Keep translations under 30 characters where possible.'
instructions: 'Legal copy. Use formal register. Preserve all placeholder variables.'
```

### `files`

**Type:** `string | string[]`

Glob pattern(s) that determine which files this context applies to. Uses the same `[locale]` placeholder as the top-level `files` option.

```ts
files: ['src/locales/marketing/[locale].json']
files: ['src/locales/legal/[locale].json', 'src/locales/compliance/[locale].json']
```

### `glossary`

**Type:** `string`

Path to a JSON glossary file. Glossary terms are included in the LLM prompt to ensure consistent translation of domain-specific terminology.

```ts
glossary: './glossaries/legal.json'
```

## Glossary files

A glossary file is a JSON object mapping source terms to their expected translations per locale:

```json
{
  "terms": {
    "Data Controller": {
      "es": "Responsable del tratamiento",
      "fr": "Responsable du traitement",
      "de": "Verantwortlicher",
      "ja": "データ管理者"
    },
    "Personal Data": {
      "es": "Datos personales",
      "fr": "Données personnelles",
      "de": "Personenbezogene Daten",
      "ja": "個人データ"
    },
    "Consent": {
      "es": "Consentimiento",
      "fr": "Consentement",
      "de": "Einwilligung",
      "ja": "同意"
    }
  }
}
```

When a glossary is attached to a context, koto sends the relevant terms to the LLM as reference, ensuring translations use the correct domain-specific terminology.

## Translating a specific context

You can translate only the files belonging to a specific context:

```bash
# Translate only legal content
npx koto translate --context legal

# Translate only marketing content
npx koto translate --context marketing
```

This is useful when you've updated only one section of your app and want fast, targeted translations.

## Best practices

1. **Start with `default`** — Define a `default` context for your general UI strings. This catches everything not matched by a specific context.

2. **Be specific with tone descriptions** — Instead of "formal," write "formal and precise, suitable for legal documents." The more specific your tone description, the better the LLM performs.

3. **Use glossaries for domain terms** — If your app has industry-specific terminology (legal, medical, financial), create glossary files to ensure consistency across all translations.

4. **Organize files by context** — Structure your locale files so that each context maps to a clear directory. This makes the `files` patterns simple and predictable.

   ```
   src/locales/
   ├── en.json              # default context
   ├── marketing/
   │   └── en.json          # marketing context
   ├── legal/
   │   └── en.json          # legal context
   └── support/
       └── en.json          # support context
   ```

5. **Keep contexts focused** — Each context should represent a distinct translation need. If two contexts have the same tone and instructions, merge them.
