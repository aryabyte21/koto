## Detailed comparisons

### koto vs Lingo.dev

**Lingo.dev** is a cloud-based AI translation service with its own proprietary translation engine.

<CardGrid>
  <Card title="Choose koto if">
    - You want full control over which LLM performs your translations
    - You need context profiles for different tones across your app
    - You want to run translations locally or in your own CI without a third-party service
    - You prefer open source (MIT) with no vendor lock-in
  </Card>
  <Card title="Choose Lingo.dev if">
    - You want a managed service with no infrastructure to set up
    - You need human review workflows built in
    - You prefer a turnkey solution over configuration
  </Card>
</CardGrid>

**Key difference:** koto lets you bring any LLM (OpenAI, Anthropic, Gemini, Ollama) while Lingo.dev locks you into their proprietary translation engine. koto's context profiles have no equivalent in Lingo.dev.

---

### koto vs Tolgee

**Tolgee** is a localization platform with a web-based translation editor, in-context translation, and API-based workflows.

<CardGrid>
  <Card title="Choose koto if">
    - Your team works primarily in code, not a web UI
    - You want AI-first translation with context awareness
    - You want to avoid running or paying for a localization platform
    - You need quality scoring and automated quality gates
  </Card>
  <Card title="Choose Tolgee if">
    - You have non-technical translators who need a visual editor
    - You want in-context editing directly in your running app
    - You need a collaborative platform with role-based access
  </Card>
</CardGrid>

**Key difference:** Tolgee is a full localization platform aimed at teams with dedicated translators. koto is a developer tool that fits into your existing code-and-CI workflow.

---

### koto vs Paraglide

**Paraglide** (by Inlang) is a compiler-based i18n library that generates type-safe message functions at build time.

<CardGrid>
  <Card title="Choose koto if">
    - You want AI-powered translation, not just a type-safe i18n runtime
    - You need translations generated automatically, not written manually
    - You want to use any LLM provider
    - You need context profiles for different parts of your app
  </Card>
  <Card title="Choose Paraglide if">
    - You want the smallest possible runtime bundle (tree-shakeable per-message)
    - You have human translators and just need type safety for the dev workflow
    - You're already in the Inlang ecosystem
  </Card>
</CardGrid>

**Key difference:** Paraglide is an i18n runtime/compiler. koto is a translation tool. They solve different parts of the problem and can even be used together — koto generates the translations, Paraglide serves them.

---

### koto vs Crowdin

**Crowdin** is an enterprise localization platform used by large teams with professional translators.

<CardGrid>
  <Card title="Choose koto if">
    - You're a developer team that wants automated AI translations
    - You don't have dedicated translators and want to ship fast
    - You want a free, open-source tool with no per-word pricing
    - You need context-aware translations without complex TMS setup
  </Card>
  <Card title="Choose Crowdin if">
    - You have a professional localization team
    - You need support for 50+ file formats
    - You need translation memory, glossary management, and QA at enterprise scale
    - You need integrations with design tools (Figma, Sketch)
  </Card>
</CardGrid>

**Key difference:** Crowdin is an enterprise TMS (Translation Management System). koto is a developer-first CLI tool. Crowdin makes sense when you have professional translators and complex workflows. koto makes sense when your dev team owns translations and wants automation.

## When to use koto

koto is the right choice when:

- **Your team writes translations in code**, not a web UI
- **You want AI to handle the translation**, not just manage human translators
- **Context matters** — different parts of your app need different translation tones
- **You want control** over which LLM does your translations (and you want to switch freely)
- **You value simplicity** — `npx koto translate` and you're done
- **CI/CD is your workflow** — translations happen in the pipeline, not a separate platform

## Migration from other tools

If you're migrating from another tool, koto works with standard locale file formats (JSON, YAML, PO, TypeScript). In most cases, migration is straightforward:

1. Point koto at your existing locale files via the `files` config option
2. Run `npx koto translate --force` to re-translate with context awareness
3. Review the output and adjust context profiles as needed

Your existing i18n runtime (next-intl, i18next, react-intl, etc.) stays the same. koto replaces only the translation step.
