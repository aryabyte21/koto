## koto vs Lingo.dev

**Lingo.dev** is a cloud-based AI translation service with a proprietary translation engine.

<CardGrid>
  <Card title="Choose koto if">
    - You want full control over which LLM performs translations
    - You need context profiles for different tones across your app
    - You want translations running locally or in your own CI
    - You prefer open source (MIT) with no vendor lock-in
  </Card>
  <Card title="Choose Lingo.dev if">
    - You want a managed service with no setup
    - You need built-in human review workflows
    - You prefer turnkey over configuration
  </Card>
</CardGrid>

**Key difference:** koto lets you bring any LLM while Lingo.dev locks you into their engine. Context profiles have no equivalent in Lingo.dev.

---

## koto vs Tolgee

**Tolgee** is a localization platform with a web editor, in-context translation, and API-based workflows.

<CardGrid>
  <Card title="Choose koto if">
    - Your team works in code, not a web UI
    - You want AI-first translation with context awareness
    - You want to avoid running or paying for a localization platform
    - You need quality scoring and automated quality gates
  </Card>
  <Card title="Choose Tolgee if">
    - You have non-technical translators who need a visual editor
    - You want in-context editing in your running app
    - You need a collaborative platform with role-based access
  </Card>
</CardGrid>

**Key difference:** Tolgee is a full localization platform for teams with dedicated translators. koto is a developer tool that fits into code-and-CI workflows.

## koto vs Paraglide

**Paraglide** (by Inlang) is a compiler-based i18n library that generates type-safe message functions at build time.

<CardGrid>
  <Card title="Choose koto if">
    - You want AI-powered translation, not just a type-safe runtime
    - You need translations generated automatically
    - You want to use any LLM provider
    - You need context profiles
  </Card>
  <Card title="Choose Paraglide if">
    - You want the smallest runtime bundle (tree-shakeable per-message)
    - You have human translators and just need type safety
    - You're in the Inlang ecosystem
  </Card>
</CardGrid>

**Key difference:** Paraglide is an i18n runtime/compiler. koto is a translation tool. They solve different parts of the problem and can be used together -- koto generates translations, Paraglide serves them.

## koto vs Crowdin

**Crowdin** is an enterprise localization platform for large teams with professional translators.

<CardGrid>
  <Card title="Choose koto if">
    - You're a dev team that wants automated AI translations
    - You don't have dedicated translators and want to ship fast
    - You want a free, open-source tool with no per-word pricing
    - You need context-aware translations without TMS complexity
  </Card>
  <Card title="Choose Crowdin if">
    - You have a professional localization team
    - You need 50+ file format support
    - You need enterprise-scale translation memory and QA
    - You need design tool integrations (Figma, Sketch)
  </Card>
</CardGrid>

**Key difference:** Crowdin is an enterprise TMS. koto is a developer-first CLI. Crowdin makes sense with professional translators and complex workflows. koto makes sense when your dev team owns translations and wants automation.

## When to use koto

- **Your team writes translations in code**, not a web UI
- **You want AI to handle translation**, not just manage human translators
- **Context matters** -- different parts of your app need different tones
- **You want control** over which LLM runs and the ability to switch freely
- **You value simplicity** -- `npx koto translate` and done
- **CI/CD is your workflow** -- translations happen in the pipeline

## Migrating from another tool

koto works with standard locale file formats. Migration is usually straightforward:

1. Point koto at your existing locale files via the `files` config
2. Run `npx koto translate --force` to re-translate with context awareness
3. Review output and adjust context profiles as needed

Your existing i18n runtime (next-intl, i18next, react-intl) stays the same. koto replaces only the translation step.
